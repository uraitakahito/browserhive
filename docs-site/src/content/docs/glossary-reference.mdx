## 1. WACZ format terms (the artifact archive format)

WACZ is a portable web-archive format that packs WARC, indexes, and metadata into a single ZIP file — BrowserHive's final artifact.
It holds 5 entries: `archive/data.warc.gz` / `pages/pages.jsonl` / `indexes/index.cdxj` / `fuzzy.json` / `datapackage.json`.

<span id="t-wacz"></span>
<span id="t-WACZ"></span>
**WACZ** (Web Archive Collection Zipped / extension `.wacz`)
A portable web-archive format that packs WARC + indexes + metadata into one ZIP file. BrowserHive's final artifact.
Contents are the following 5 entries: `archive/data.warc.gz` / `pages/pages.jsonl` / `indexes/index.cdxj` / `fuzzy.json` / `datapackage.json`.

<span id="t-WARC"></span>
**WARC** (Web ARChive)
The standard format that concatenates HTTP requests/responses as one record each.
BrowserHive writes each record as an **independent gzip member** into `archive/data.warc.gz`.

<span id="t-archivedatawarcgz"></span>
**`archive/data.warc.gz`**
The gzip-compressed entry inside the WACZ that holds the WARC records themselves. Records are concatenated as independent gzip members.

<span id="t-pagespagesjsonl"></span>
<span id="t-pagesjsonl"></span>
**`pages/pages.jsonl`**
The entry listing captured pages, one JSON per line (JSON Lines). Replay uses it as the per-page entry point.

<span id="t-indexesindexcdxj"></span>
**`indexes/index.cdxj`**
The CDXJ-format index file placed inside the WACZ (see **CDXJ** below).

<span id="t-CDXJ"></span>
**CDXJ**
The **index** of records inside the WARC: lines of `<SURT> <timestamp> <JSON{offset,length,…}>` sorted in SURT order.
Replay uses it to look up "URL → byte position in the WARC".

<span id="t-SURT"></span>
**SURT** (Sort-friendly URI Reordering)
A normalized URL with the host reversed. Entries of the same domain become adjacent, which makes it a good sort key for CDXJ.

<span id="t-comexamplepath"></span>
**SURT key (e.g. `com,example)/path`)**
The SURT-normalized key notation. Reversing the host of `https://example.com/path` yields `com,example)/path`, which sorts as the CDXJ line prefix.

<span id="t-fuzzyjson"></span>
**`fuzzy.json`**
Replay rules that strip cache-buster queries (`?_=…`, etc.), so URLs whose values change every load still match the recorded response.

<span id="t-datapackagejson"></span>
**`datapackage.json`**
The Frictionless descriptor. It carries the `sha256:<hex>` and byte counts of the other resources in the WACZ, anchoring tamper detection.

<span id="t-sha256"></span>
**`sha256:…` (payload digest encodings)**
The prefixed notation of a SHA-256 hash. WARC records use `sha256:<base32>` and `datapackage.json` uses `sha256:<hex>` —
**the same value in two encodings** (see **payload digest** below).

<span id="t-WARCTypemetadata"></span>
<span id="t-applicationwarcfields"></span>
<span id="t-resourceType"></span>
<span id="t-blockedReason"></span>
<span id="t-status"></span>
**metadata record** (`WARC-Type: metadata`)
A WARC record whose body is `application/warc-fields`. It records failed / incomplete / blocked exchanges.
The body carries the `resourceType` / `blockedReason` / `status` fields (the "plan 3" extension).

- `WARC-Type: metadata` — the WARC header marking this record as the metadata kind.
- `application/warc-fields` — the body MIME type of a metadata record (a set of `key: value` lines).
- `resourceType` — the kind of resource being recorded.
- `blockedReason` — why the exchange was blocked.
- `status` — the fetch outcome status of that resource.

<span id="t-payloaddigest"></span>
**payload digest**
The SHA-256 of a record body. WARC holds it as `sha256:<base32>`, `datapackage.json` as `sha256:<hex>` —
**the same value in two encodings**.

<span id="t-gzipmember"></span>
**gzip member**
A gzip block that gunzips on its own. Members remain valid gzip when concatenated and support byte seeking by offset (CDXJ relies on this).

<span id="t-replay"></span>
**replay**
"Playing back" a saved WACZ to display the page as it was (ReplayWeb.page / wabac.js, etc.). BrowserHive is the recording side — the producer.

:::tip[On wording]
The **spelling discipline** for WACZ spec terms like `ZIP file` and `WARC record` (e.g. never write `zip`) is
collected in [WACZ vocabulary](/wacz-vocabulary/). Definitions live here; word choice lives there.
:::

## 2. XState API terms (state machines, v5)

BrowserHive's worker control is written with XState v5 state machines. This section makes the **API terms that appear in the code** easy to look up.
To learn the concepts, see [XState primer + the features BrowserHive uses](/xstate-primer/); for how they are used in practice, see
[Worker spawn & loop](/worker-spawn-and-loop/).

<span id="t-statemachine"></span>
**state machine**
A model that declares "which **state** we are in" and "which **event** moves us **where**". It structurally rules out impossible state combinations.

<span id="t-actor"></span>
**actor**
A running machine instance. Obtain one with `createActor(machine).start()`, drive it with `send`, and peek with `getSnapshot`. XState v5 is actor-centric.

<span id="t-createActor"></span>
**`createActor(machine).start()`**
Creates an **actor** from a machine and starts it. Send events to the returned actor with `send` and read the current state with `getSnapshot`.

<span id="t-send"></span>
**`send`**
The method that delivers an event to an actor to drive the machine. Pass an event object like `{ type: "CONNECT" }`.

<span id="t-getSnapshot"></span>
**`getSnapshot`**
The method that returns the actor's current **snapshot** (see **snapshot** below).

<span id="t-state"></span>
**state**
An exclusive mode. atomic (simple) / **compound** (nested, e.g. `active{running,degraded}`) / **final** (`type:"final"`, terminal).

<span id="t-event"></span>
**event**
A "something happened" notification delivered to the machine — a discriminated union like `{ type: "CONNECT" }`.

<span id="t-transition"></span>
**transition**
The rule "in this state, on this event → that state": `on: { EVENT: target }`. Written as an array, guards select the branch.

<span id="t-context"></span>
**`context`**
Accompanying data that state names cannot express (counters, error history, current task, …). Update with `assign`; read from a snapshot via `.context`.

<span id="t-guard"></span>
**`guard`**
A condition deciding whether a transition may fire. With multiple transitions in an array, **the first branch whose guard passes** wins (e.g. `canRetry`).

<span id="t-action"></span>
<span id="t-assign"></span>
**`action` / `assign`**
**`action`** = a synchronous side effect on transition (logging, …). **`assign`** = the special action that updates `context`.

<span id="t-invoke"></span>
<span id="t-onDone"></span>
<span id="t-onError"></span>
**`invoke`**
Starts another actor tied to a state. It runs **only while in that state** and stops automatically on exit.
Branch on the result with `onDone` / `onError` (`onDone` receives normal completion, `onError` receives failure).

<span id="t-spawn"></span>
**`spawn`**
Dynamically creates multiple actors and keeps them in `context`. You manage their lifetime yourself (BrowserHive `spawn`s its worker set).

<span id="t-fromPromise"></span>
**`fromPromise`**
An async actor that resolves/rejects once and ends. Received via `onDone` (`event.output`) / `onError`.

<span id="t-fromCallback"></span>
<span id="t-sendBack"></span>
<span id="t-receive"></span>
**`fromCallback`**
A resident actor. It sends events up with `sendBack`, receives parent instructions with `receive`, and cleans up in the returned function.

- `sendBack` — sends an event **child → parent** from a fromCallback actor.
- `receive` — receives **parent → child** messages inside a fromCallback actor.

<span id="t-tags"></span>
<span id="t-hasTag"></span>
**`tags` / `.hasTag()`**
`tags` are labels attached to states (`"healthy"`, …). `snapshot.hasTag("healthy")` lets you **test a property without knowing state names**.

<span id="t-snapshot"></span>
<span id="t-value"></span>
<span id="t-matches"></span>
**snapshot**
A snap of the actor's current state. It carries `.value` (current state name) / `.context` / `.matches()` (state test) / `.hasTag()` / `.can()`.

- `.value` — the current state name.
- `.matches()` — tests whether the machine is in the given state.

<span id="t-final"></span>
**`"final"`**
The kind that declares a state terminal (`type: "final"`). When every child of a compound state reaches `final`, the parent's `onDone` fires.

<span id="t-setup"></span>
**`setup()`**
v5's typed parts box. Register `types` / `actors` / `guards` / `actions` first, then reference them by name inside `createMachine`.
